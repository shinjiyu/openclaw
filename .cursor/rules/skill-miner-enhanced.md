# 进化 Agent 增强指引

本规则供 skill-miner 在从日志中提炼 Skill/Rule 时遵循。

---

## 1. 置信度标注

在 Skill/Rule 的 Reference 中必须标注：

| 标注类型 | 格式 | 说明 |
|----------|------|------|
| 会话数量 | `基于 N 次会话提炼` | N=1 时需显式说明 |
| 验证状态 | `待多项目验证` / `已在 N 个项目验证` | 单日志来源时使用前者 |
| 证据追溯 | `来源日志：<session-id>` | 便于后续交叉验证 |

**禁止**：单日志提炼时过度泛化，不标注免责说明。

---

## 2. 抽象层次一致性

### 2.1 优先级逻辑统一

- 提案阶段的优先级应与实施阶段一致
- 若实施时调整优先级，必须在文档中说明理由

### 2.2 领域 vs 通用边界

明确标注每个模式属于：

- **领域特定**：仅适用于特定技术栈 → 放入 Skill
- **可抽象通用**：适用于多种场景 → 考虑 Rule

---

## 3. 提案与实施映射

### 3.1 映射表要求

实施报告中必须包含提案 ID → 最终文件的对应关系。

### 3.2 合并决策说明

若将多个提案合并为单一文件，需说明：
- 合并理由
- 合并后触发条件是否过宽

---

## 4. Agent 文件结构合规检查

增量更新 `.cursor/agents/*.md` 时：

1. **编辑前**：先 `Read` 全文，检查是否已存在重复 frontmatter（多段 `---` YAML 块）
2. **若发现重复**：本次编辑必须一并合并/去重，不能仅做增量追加
3. **单 frontmatter 原则**：每个 agent 文件仅允许一段 frontmatter，位于文件开头

## 5. 脚本化落地判定

当一个模式满足「步骤确定、参数可配置、重复执行 ≥2 次」时，优先建议固化为可执行脚本而非仅写 Rule：

- Rule 适合指导 Agent 认知和决策（"何时做"、"为什么"）
- Script 适合替代 Agent/人的重复手动执行（"怎么做"、"自动化"）
- 两者可共存：Rule 引导 Agent 在合适时机使用脚本

**反模式**：将确定性操作流程仅写为 Rule，导致 Agent 每次仍需逐步手动执行（如部署验证规则存在但无部署脚本，Agent 每次仍要手工 SSH + git pull + build + restart）。

---

## 6. 输出间一致性检查

创建或更新 Skill/Rule 前，必须检查：

- [ ] 与已有 Rule 是否矛盾
- [ ] 与已有 Rule 是否重复
- [ ] 同一模式在不同输出中优先级是否一致

### 矛盾处理规则

| 矛盾类型 | 处理方式 |
|----------|----------|
| 技术建议矛盾 | 以更保守一方为准，或标注「需实测」 |
| 规则重复 | 由单一规则主导，另一方改为引用 |
| 优先级不一致 | 说明理由并统一 |

---

## 7. 与归纳 Agent 的接口约定

### 6.1 输入使用方式

当输入包含 log-analyzer 报告时：
1. 引用 log-analyzer 的问题编号（P1–P3、W1–W5）
2. 遗漏项评估是否补充

### 6.2 输出追溯要求

提案应包含来源标注，形成双向追溯。

### 6.3 重叠模式去重

对双方均识别的模式，由单一真相源主导，另一方引用。

---

## 8. Subagent 提案落地机制

| 条件 | 动作 |
|------|------|
| 首次出现 | 标记「待观察」，不立即创建 |
| N 次重复（N≥2） | 可创建，标注「基于 N 次会话提炼」 |
| 用户明确要求 | 直接创建，标注「用户指定」 |

---

## 9. 引用前验证

在建议「由 X 处理」或「遵循 X 规则」前：
1. 搜索 `.cursor/rules/`、`.cursor/skills/` 确认是否存在
2. 若存在，引用而非重复创建
3. 若不存在，创建或标注「建议创建」
